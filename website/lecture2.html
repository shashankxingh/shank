Lecture 2: How a Shank Program Runs

In the previous lecture, we learned how to plan our logic and wrote our first basic Shank program. But what actually happens when you ask the computer to run your Shank code? It isn't magic! In this lecture, we will peek under the hood and understand the lifecycle of a Shank program, from plain text to a native executable.

1. The Compilation Pipeline

Shank is a compiled language. Unlike interpreted languages (such as Python or JavaScript) which read and execute source code line-by-line at runtime, Shank translates your entire code into machine language before it ever runs. This translation process is called Compilation, and it happens in several distinct stages:

Stage A: Lexical Analysis (Lexing)

First, the compiler reads your source code (the .shnk file) as a massive string of characters. The Lexer breaks these characters down into meaningful chunks called tokens. For example, the code let x = 5 is converted into tokens like [KEYWORD_LET], [IDENTIFIER(x)], [EQUALS], [NUMBER(5)].

Stage B: Parsing (Syntax Analysis)

The parser takes the stream of tokens and groups them into a structure called an Abstract Syntax Tree (AST). This tree represents the grammatical structure of your program. If you forget to indent properly, or if you write let = x 5, the parser will fail and throw a Syntax Error.

Stage C: Semantic Analysis

Once the tree is built, the compiler checks the meaning of the code. It verifies types (e.g., ensuring you don't try to divide a string by an integer) and checks if variables are declared before they are used. Shank is strictly typed, so this phase is critical for safety.

Stage D: Code Generation (NASM)

If your code passes all checks, the Shank compiler generates Assembly Code. Specifically, it targets the x86-64 architecture and generates NASM (Netwide Assembler) syntax. Assembly language is a low-level programming language that is very close to machine code.

Stage E: Assembling and Linking (GCC)

Finally, an assembler converts the NASM code into object files (binary code), and a linker (like GCC) combines your object files with the standard library to create the final .exe (on Windows) or ELF binary (on Linux).

2. Static vs Dynamic Typing

A crucial aspect of how Shank operates is its type system. Shank uses Static Typing. This means that the types of all variables are known at compile-time (before the program runs).

# Valid Shank code
mut age: int = 20
age = 21

# Compile-time ERROR! 
# You cannot assign a string to an integer variable.
age = "twenty-two"

In contrast, dynamically typed languages (like Python) figure out types on the fly during execution, which can lead to unexpected crashes while the program is running.

3. Memory Layout: Stack vs Heap

When you run the compiled executable, the operating system allocates memory for your program. Shank manages two primary areas of memory:

  • The Stack: Fast, organized, and limited. Local variables and function calls are stored here. When a function finishes, its stack memory is automatically cleared.
  • The Heap: Large, flexible, but slower. Used for dynamic data structures whose size might change at runtime (like large dynamic arrays or complex objects). Memory here must be managed carefully.

4. Compile-Time vs Run-Time Errors

As a programmer, you will encounter bugs. It is helpful to categorize them:

  • Compile-Time Errors: The compiler refuses to generate an executable. Examples include syntax errors (wrong indentation), type mismatches, or using undeclared variables. These are "good" errors because they are caught early.
  • Run-Time Errors: The code compiles perfectly, but crashes when you run it. Examples include dividing by zero, or trying to access the 100th element of a 5-element array. These are harder to debug.

5. Summary

Writing Shank code is just the first step. The compiler acts as a strict translator, ensuring your code is structurally and logically sound before handing it over to the assembler and linker. Understanding this pipeline will make you much better at deciphering error messages and writing efficient code.
