Lexical & Execution Model

A formal specification of Shank's syntax boundaries and compiler pipeline.

Lexical Structure

Shank source files must be encoded in UTF-8. The lexer processes the source code into a stream of tokens, discarding whitespace and comments while preserving indentation semantics.

1. Indentation

Like Python, Shank uses whitespace indentation to define code blocks (scope) instead of braces `{ }`. An increase in indentation generates an `INDENT` token, and a decrease generates a `DEDENT` token. Mixing spaces and tabs on the same line is strictly prohibited and will trigger a lexer error.

A standard indentation level is typically 4 spaces.

2. Comments

Comments begin with the hash `#` character and extend to the end of the physical line.

# This entire line is ignored by the lexer
im int x = 5  # This is an inline comment

3. Keywords

The following identifiers are reserved words and cannot be used as variable or function names:

  • im, mut
  • int, float, str, bool
  • when, elif, otherwise
  • repeat, as
  • fn, return
  • and, or, not
  • true, false
  • out, outt, outc

The Data Model

Shank provides four primitive data types. Variables are strictly typed at compile-time.

Type Keyword Memory Representation Example
Integer int 64-bit signed integer (QWORD) 42
Floating-point float 64-bit IEEE 754 double-precision 3.14159
Boolean bool 8-bit unsigned integer (0 = false, 1 = true) true
String str 64-bit pointer to a null-terminated UTF-8 byte array "hello"

Execution Pipeline

Shank is not interpreted. It follows a rigorous compilation pipeline directly to machine code.

  1. Lexing: `src/lexer.c` tokenizes the source text.
  2. Parsing: `src/parser.c` constructs an Abstract Syntax Tree (AST) using Recursive Descent.
  3. Type Checking: `src/checker.c` traverses the AST, resolving variable scopes, enforcing `im` mutability constraints, and validating operator type safety.
  4. Code Generation: `src/codegen.c` performs a single-pass translation of the verified AST into NASM-compatible x86-64 assembly instructions.
  5. Assembling & Linking: The compiler invokes the bundled `nasm` assembler to create `.obj` files, and then links them against the C runtime using `gcc` to produce a standalone executable.
Memory Architecture Insight: During the Code Generation phase, the compiler dynamically calculates the maximum depth of nested variables within a function. It then emits a single stack reservation instruction (e.g., sub rsp, 280) at the function prologue, ensuring highly efficient and robust memory allocation without dynamic heap overhead.
