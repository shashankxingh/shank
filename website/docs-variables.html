Variables & Mutability

An exhaustive deep-dive into how Shank allocates, scopes, and protects memory.

Unlike dynamic languages that allow variables to change types or be overwritten arbitrarily, Shank enforces strict memory mutability. This guarantees that data you expect to remain constant cannot be accidentally overwritten, preventing a massive class of runtime logic bugs.

1. Immutable Variables (im)

By default, all data in Shank should be declared as Immutable using the im keyword. Once an im variable is allocated and assigned a value on the stack, that memory address is locked. It can never be reassigned.

Basic Syntax

im int limit = 100
im str name = "Shank"

Strict Typing

Shank's Type Checker (`src/checker.c`) strictly requires explicit type declarations for all variables. There is no type inference.

im int limit = 100
im bool is_active = true

The Protection Mechanism

If you attempt to assign a new value to an `im` variable, the compilation will fail immediately with a fatal error. The compiler will refuse to generate x86-64 assembly.

im int days = 7
days = 8  # FATAL: Cannot assign to immutable variable 'days'

2. Mutable Variables (mut)

If an algorithm fundamentally requires modifying memory in place (for example, accumulating a sum in a loop), you must explicitly opt-in to mutability using the mut keyword.

mut int total = 0

repeat 5 as i:
    total = total + i  # Perfectly valid, total is mutable.

outc "Total is "
out total
Best Practice: Always default to im. Only upgrade a variable to mut if the compiler complains when you attempt to reassign it. This minimizes side-effects and makes code highly predictable.

3. Input Assignment via put

When requesting user input using the put keyword, the Type Checker performs an advanced backward-resolution to determine what type to cast the input into.

im int age = put "Enter your age: "

In the above example, because `age` is strictly typed as an `int`, the compiler automatically casts the raw string input from the user into a 64-bit integer. You must always provide a type for variables accepting input.

im str name = put("Enter name: ")

4. Scope & Shadowing Rules

Variables in Shank are block-scoped. When an indentation block (`when`, `repeat`, `fn`) terminates, all variables declared within that block are popped off the symbol table.

Shadowing Prohibition

To prevent extremely difficult-to-track bugs, Shank forbids variable shadowing within the same scope. You cannot declare a variable with the same name twice in the same block.

im x = 10
im x = 20 # FATAL: Variable 'x' already defined in this scope

5. Under the Hood: The Compiler Backend

When you declare an `im` or `mut` variable, what does Shank actually do with your CPU and RAM? Let's look at the generated x86-64 assembly in `src/codegen.c`.

The Symbol Table to Stack Frame Translation

During the AST traversal, the Type Checker stores the variable in a Hash Map (the Symbol Table) and assigns it a "stack offset" (e.g., `-8`, `-16`).

During the Code Generation phase, Shank emits instructions that allocate space on the Base Pointer (`rbp`) stack frame. All `int`, `float`, and pointer variables require exactly 8 bytes (a QWORD) on 64-bit systems.

For example, compiling im int limit = 100 yields the following raw instructions:

mov rax, 100       ; Load 100 into the rax register
mov [rbp-8], rax   ; Store the value onto the stack at offset -8

Because `limit` was declared as `im`, the C-based Type Checker refuses to emit any subsequent `mov [rbp-8], ` instructions, completely locking the memory safely at compile time!
