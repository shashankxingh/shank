The Shank Tutorial

A comprehensive introduction to writing, compiling, and running Shank.

Shank is an exceptionally lean, statically typed programming language designed to strip away the boilerplate and complexity of modern systems programming. With an indentation-based syntax heavily inspired by Python and a custom-built x86-64 backend that emits raw Windows assembly, Shank is built for raw performance without sacrificing developer ergonomics.

This tutorial will guide you through setting up your environment, understanding Shank's execution semantics, and writing robust programs.

1. Installation and Setup

Shank requires no runtime environments, no interpreters, and no heavy SDKs. The compiler itself is a lightweight, zero-dependency executable.

Using the Portable Release

The easiest way to get started is by downloading the standalone `.zip` distribution.

  1. Download the `shank-v0.1-win64.zip` from the homepage.
  2. Extract the archive to a permanent directory (e.g., `C:\shank`).
  3. Add `C:\shank` to your System's `PATH` environment variable.
  4. Open a new terminal and type shankc help. If the help menu appears, your installation is successful.
Note: The portable release contains shankc.exe along with deeply integrated, stripped-down versions of the MinGW-w64 GCC toolchain and NASM. These are required by the backend to correctly assemble and link the generated x86-64 machine code without forcing you to install Microsoft Visual C++ Build Tools.

2. The Compiler CLI

The `shankc` command-line interface provides everything you need to check, compile, and run your code.

  • shankc run <file.sk>: Compiles the source file, executes the binary, and automatically cleans up the generated `.exe` after the process exits. This is recommended for local development.
  • shankc build <file.sk> [-o output]: Compiles the source code into a permanent `.exe` executable file ready for distribution.
  • shankc emit <file.sk> [-o output.asm]: Stops compilation before the assembly phase and emits the raw NASM-compatible x86-64 assembly instructions to a file. Useful for debugging compiler backend logic.
  • shankc check <file.sk>: Runs the Lexer, Parser, and Semantic Analyzer (Type Checker) without emitting code. Used for rapid syntax validation.

3. Hello, World!

Let's write our first Shank program. Create a file named `hello.sk`.

# This is a comment in Shank
out "Hello, World!"

To run this code, execute the following command in your terminal:

$ shankc run hello.sk
Hello, World!

Notice that there are no `main` functions, no class wrappers, and no semicolons. Shank evaluates statements from the top of the file downward, wrapping them automatically into a hidden `shank_main` entry point during assembly generation.

4. Variables and Strict Mutability

Shank is statically typed, meaning variable types are strictly enforced at compile time. However, unlike Python, Shank places a heavy emphasis on memory mutability correctness. All variables must be explicitly declared as either Immutable (`im`) or Mutable (`mut`).

Immutable Variables

By default, variables should be declared as `im`. Once an `im` variable is assigned a value in memory, it can never be altered. Attempting to assign a new value to an `im` variable results in a fatal compile-time error.

im int limit = 100
out "The limit is {limit}"

# Compile Error: Cannot assign to immutable variable 'limit'
limit = 200

Mutable Variables

If you intend to modify the memory address of a variable (such as a counter in a loop), you must explicitly flag it with `mut`.

mut int counter = 0

repeat 5 as i:
    counter = counter + i

out counter // Prints 10
Memory Architecture: Under the hood, both `im` and `mut` variables are allocated to the x86-64 stack frame via the Base Pointer (`rbp`). The difference is purely semantic and enforced heavily by the `src/checker.c` Type Analyzer during the AST traversal phase.
