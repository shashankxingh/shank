Functions

An exhaustive deep-dive into Shank's function mechanics, calling conventions, and stack frame generation.

Functions in Shank are pure, strictly typed logical boundaries. Shank implements the native Microsoft x64 ABI (Application Binary Interface) calling convention directly, allowing Shank functions to be extremely fast and theoretically interoperable with native C code.

1. Defining Functions

Functions are defined using the fn keyword. You must declare the types of all arguments. The compiler accepts both modern Python-style annotations (`name: type`) and classic C-style prefixes (`type name`).

Basic Syntax

fn add(a: int, b: int) -> int:
    return a + b

Return Semantics

The -> operator defines the return type. If a function does not return a value, you can omit the arrow entirely, or explicitly return none. Every branch of a function that returns a value must explicitly terminate with a return keyword.

fn greet(name: str):
    out "Hello, {name}!"
    # Implicit return

2. Calling Functions

Functions are invoked using standard parenthesis notation. Shank enforces strict arity (argument count) and strict type matching during the Semantic Checking phase.

im int total = add(10, 20)
Type Enforcement: You cannot pass a str to an int argument. Attempting to call add("10", 20) will trigger a compiler error before assembly generation begins.

3. Under the Hood: The x64 Calling Convention

To understand why Shank functions are so fast, we have to look at the assembly emitted by `src/codegen.c`. Shank does not build a heavy virtual machine call stack; it talks directly to the CPU registers.

The Prologue and Epilogue

When you define a function, Shank immediately generates an assembly label prefixed with sk_, followed by the classic C-style stack prologue.

global sk_add
sk_add:
    push rbp
    mov rbp, rsp
    sub rsp, 000032  ; Dynamically patched local stack reservation

When return is called, the compiler emits the epilogue, instantly destroying the stack frame and passing control back to the caller.

    mov rsp, rbp
    pop rbp
    ret

Register Argument Passing

To minimize slow RAM access, Shank adheres to the Microsoft x64 fast-call convention. The first four arguments are passed directly into CPU registers, not the stack:

  • Argument 1: rcx
  • Argument 2: rdx
  • Argument 3: r8
  • Argument 4: r9

If you call `add(10, 20)`, the compiler generates mov rcx, 10 and mov rdx, 20, and then executes call sk_add.

Return Values

Return values are always passed back through the rax register. When evaluating `a + b` in the example above, the result is computed into `rax`, and the `ret` instruction leaves it there for the caller to immediately utilize.
