Control Flow (Conditionals)

An exhaustive deep-dive into Shank's dual conditional syntax, boolean rigidity, and AST evaluation mechanics.

Unlike dynamic languages that use "truthy" concepts (like treating empty lists or zero as false), Shank enforces strict binary logic. Every branch condition must evaluate mathematically to a `bool`. The compiler directly generates highly optimized Jump instructions (`je`, `jne`, `jmp`) based on these states.

1. The Primary Syntax: when / otherwise

Shank's preferred, idiomatic syntax for logical branching utilizes the when and otherwise keywords. This syntax cleanly describes state-based execution, discarding ambiguous `else if` chains.

Basic Evaluation

im bool system_ready = true

when system_ready:
    out "Booting the system..."
otherwise:
    out "System is offline. Halting."
No 'elif' Equivalent for When: The `when` statement strictly operates as an A/B switch. It parses via `NODE_WHEN` in `src/parser.c` directly into a single body and a single `otherwise` fallback block. If you require complex multi-step branching chains, use the secondary syntax instead.

2. The Secondary Syntax: if / elif / else

For scenarios requiring deeply chained conditions, Shank implements the traditional C/Python-style branching logic.

im int score = 85

if score >= 90:
    out "Grade: A"
elif score >= 80:
    out "Grade: B"
elif score >= 70:
    out "Grade: C"
else:
    out "Grade: F"

How NODE_IF Operates

During the Code Generation phase (`src/codegen.c`), the compiler linearizes these branches. It emits a test evaluation (`cmp`), followed immediately by a Jump If Equal/Not Equal (`je`/`jne`) targeting dynamically generated Assembly labels (e.g., `.L_elif_1`, `.L_else`). At the end of every successful block, a definitive Jump (`jmp .L_end`) completely escapes the chain.

3. Strict Boolean Rigidity

Shank completely eliminates the concept of "truthy" or "falsy" values at the compiler level. 

im int users = 5

# FATAL: The condition must evaluate to a bool, not an int.
if users:
    out "There are users online."

# CORRECT: Evaluate the relational math to yield a bool.
if users > 0:
    out "There are users online."

This is strictly enforced inside `src/checker.c`. The semantic analyzer explicitly checks the `resolved_type` of the condition expression. If `cond->resolved_type != type_bool`, the compilation instantly halts.

4. Nested Branching and Scope

Because Shank is indentation-based (via the `TOK_INDENT` and `TOK_DEDENT` lexer tokens), you can infinitely nest conditionals. Any variables declared within a branch are rigidly scoped to that indented block and destroyed when the block is dedented.

im bool auth = true
im int clearance = 5

if auth:
    im str local_token = "X910"
    if clearance >= 5:
        out f"Token {local_token} accepted."
    else:
        out "Clearance too low."

# 'local_token' is no longer in the symbol table here.
