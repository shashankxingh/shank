Lecture 5: Variable Declarations

Imagine you have a giant warehouse with millions of unmarked boxes. Finding a specific item would be impossible. To fix this, you put labels on the boxes so you can easily store and retrieve items. In programming, the computer's memory is that giant warehouse, and Variables are the labels you put on memory locations to store and retrieve data.

Unlike other mainstream languages, Shank takes a completely fresh and readable approach to variables, designed specifically for safety and clarity.

Topic 1: What is a Variable in Shank?

A variable is a named container that holds information that your program can use, modify, and reference. When you create a variable, the computer reserves a chunk of memory and attaches your chosen name to it.

Subtopic 1.1: The Philosophy of Safety

Shank is designed to be safe and predictable. Therefore, by default, data in Shank should be clearly marked based on whether it is allowed to change. We call this concept Mutability. Shank provides two dedicated keywords for declaring variables: im and mut.

Topic 2: Immutable Variables (im)

If you want to store a value that should never change throughout its lifespan, you use the im keyword (short for immutable). This is great for data that is set once and read many times, such as a person's date of birth or a maximum limit.

Subtopic 2.1: Declaring with `im`

The syntax for declaring an immutable variable is straight to the point: im [datatype] [name] = [value].

im int max_users = 100
im str greeting = "Welcome to Shank!"

# The following line will cause a Compile-Time ERROR because max_users is immutable!
# max_users = 101

Topic 3: Mutable Variables (mut)

If you know that a variable's value needs to be updated or changed later (like a player's score, a loop counter, or a bank balance), you must explicitly declare it as mutable using the mut keyword.

Subtopic 3.1: Declaring with `mut`

The syntax is identical to im, but you use mut instead.

mut int player_score = 0
player_score = 10  # Perfectly valid!
player_score = 15  # Still valid!

Topic 4: Naming Rules and Conventions

You cannot name your variables whatever you want. The Shank compiler enforces strict rules to keep your code readable and distinct.

Subtopic 4.1: Strict Compiler Rules

  • Must start with a letter (a-z, A-Z) or an underscore (_).
  • Cannot start with a number (e.g., `1st_player` is invalid).
  • Can only contain letters, numbers, and underscores.
  • Cannot be a reserved Shank keyword (e.g., you cannot name a variable `im`, `mut`, `fn`, or `out`).

Subtopic 4.2: Naming Conventions (Best Practices)

In Shank, the standard convention for variable names is snake_case. This means all lowercase letters, with words separated by underscores.

  • Good: `user_age`, `total_price`, `is_game_over`
  • Bad: `UserAge`, `totalPrice`, `isgameover`
